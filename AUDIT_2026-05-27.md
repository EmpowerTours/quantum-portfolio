## Original executive summary — historical, superseded

1. **The SLH-DSA variant is mislabelled everywhere.** `quantcrypt.SMALL_SPHINCS` produces pk=64, sk=128, sig=29792 → definitively **SLH-DSA-SHAKE-256s** (Level-5, 256-bit security). Code, README, SECURITY.md, SUBMISSION.md, and the Streamlit UI all say "SLH-DSA-SHAKE-**128s**". The crypto is real and in fact *stronger* than advertised, but the wrong label invites a "they don't know what they shipped" critique. **BLOCKER.** *(Empirically verified 2026-05-27.)*
2. **CI does not test the shipped crypto.** `.github/workflows/test.yml` installs `dilithium-py` (not in `requirements.txt`) and runs `tests/test_pq_signing.py`, which imports `quantcrypt`. SUBMISSION.md claims "GitHub Actions runs the full suite" — currently false. **BLOCKER.**
3. **`append_audit` is racey.** Read-prev-hash then append, no `flock`. Two concurrent Sign clicks break the chain irrecoverably with no app-level recovery path. **BLOCKER.**
4. **The liboqs-python 0.15.0 claim (2026-05-15) is false.** Verified against GitHub releases: latest liboqs-python is **0.12.0 (2025-01-16)**. Both the version number AND the date are wrong. **HIGH.** *(Discovered during audit verification 2026-05-27.)*
5. **AI forecaster has a 21-day lookahead leak.** Walk-forward training lets the last training row's `y` use prices through `as_of + 21`. "No lookahead" claim in docs is false as written. **HIGH.**

## Original blockers — historical, superseded

### B1. Wrong SLH-DSA variant labelled in docs and code

- **Files & lines**
  - `src/pq_signing.py:53` — `from quantcrypt.dss import MLDSA_65, SMALL_SPHINCS`
  - `src/pq_signing.py:75` — `SLH_DSA_ALGORITHM = "SLH-DSA-SHAKE-128s (NIST FIPS 205)"`
  - `src/pq_signing.py:8-10` — module docstring "**SLH-DSA-SHAKE-128s** (NIST FIPS 205)"
  - `app.py:333` — `col2.metric("SLH-DSA-128s (FIPS 205)", …)`
  - `README.md:17`, `SUBMISSION.md:88`, `SECURITY.md:27` — all say "SLH-DSA-SHAKE-128s"
- **Empirical verification 2026-05-27**: `SMALL_SPHINCS().keygen()` returns pk=64 B, sk=128 B; sign produces 29792 B → matches FIPS 205 SLH-DSA-SHAKE-256s parameters exactly.
- **FIPS 205 parameter tables** (public): SLH-DSA-SHAKE-128s has pk=32, sk=64, sig≈7.8 KB. Our shipped sizes match **256s** not 128s.
- **Patch**
  - Rename `SLH_DSA_ALGORITHM` to `"SLH-DSA-SHAKE-256s (NIST FIPS 205)"`, propagate string through README/SECURITY/SUBMISSION/app.py.
  - Reframe the hedge as Level-5 hash-based, not Level-1.
  - Add a test that pulls the PQClean variant name from `quantcrypt` and asserts it matches the constant.

### B2. CI does not run the actual signing code

- **Files**: `.github/workflows/test.yml:24-32`
- The "test" job runs `pip install dilithium-py` then `python tests/test_pq_signing.py`. The test file imports `src.pq_signing`, which imports `quantcrypt`. `quantcrypt` is not installed by the CI step.
- SUBMISSION.md `line 152` claims "GitHub Actions runs the full suite on Python 3.11 and 3.12 on every push." Currently false.
- **Patch**: replace `pip install dilithium-py` with `pip install -r requirements.txt`. Add `python tests/test_monad_tx.py` to the same step.

### B3. Non-atomic `append_audit` plus a Streamlit "Sign" button is a one-click chain-DoS

- **Files**: `app.py:411`, `src/orders.py:271-289`
- `append_audit` reads the last line, hashes it, then opens the log in append mode and writes. No `flock`, no atomic rename. Two concurrent calls compute the same `prev_hash`, append two entries pointing to the same predecessor, and `verify_audit_chain` permanently fails from that point.
- **Patch**: wrap the read-prev-hash + append-line block in `fcntl.flock(fh, LOCK_EX)`. Add a threaded race test.

### H2. AI-forecaster lookahead leak

- **File**: `src/ai_forecast.py:53-74` (`_train_one_asset`)
- `fwd = prices.pct_change(horizon_days).shift(-horizon_days)`; `train = df.loc[df.index <= as_of]` keeps rows whose `y` depends on prices through `as_of + 21` — strictly future.
- README `:13`, SUBMISSION `:73-74`, `src/ai_forecast.py:1-6` all claim "no lookahead" — currently false.
- Equal-weight (Sharpe 1.87) already beats AI (Sharpe 1.48), so the leak doesn't produce an alpha claim. But the technical claim is wrong and any reviewer who reproduces with correct cutoff gets a different number.
- **Patch**: `train = df.loc[df.index <= (as_of - pd.Timedelta(days=horizon_days))]`. Add regression test. Re-run backtest. Update reported Sharpe.

### H3. Audit log is the central artefact — and it's gitignored

- `.gitignore:8-15` allow-lists `signed_orders.json`, `unsigned_monad_tx.json`, charts, etc., but not `audit_log.jsonl`.
- SUBMISSION.md `:111-114` and SECURITY.md `:51-61` describe the hash-chained log as the deliverable proving tamper-evidence offline. Reviewers can't audit it — it isn't in the repo.
- **Patch**: add `!outputs/audit_log.jsonl` to `.gitignore`, regenerate clean, commit. Add CI step running `verify_audit_chain`.

### H4. Hash-chain "last line" finder has a silent 64-KB invariant

- **File**: `src/orders.py:253-268` — `_last_line_hash` reads only the trailing 64 KB.
- Current entries: 47904 and 47871 bytes. Safe today.
- SLH-DSA-256s `s` → `f` swap or Falcon addition crosses 64KB and `_last_line_hash` returns the SHA-256 of a truncated trailing fragment.
- **Patch**: chunked reverse-line iterator. Or sidecar `last_line_hash` file written atomically with each append.

### H5. `monad_tx.encode_order_calldata` ≠ `pq.canonical_bytes`

- `src/pq_signing.py:99-108` — `ensure_ascii=False`
- `src/monad_tx.py:90-91` — default `ensure_ascii=True`
- Non-ASCII pool labels desync the byte string a Solidity / non-Python verifier would hash.
- **Patch**: in `encode_order_calldata`, replace `json.dumps(...)` with `pq.canonical_bytes(signed.order.to_dict())`.

### H6. `verified_at_sign_time` is recorded but never gates the append

- **File**: `src/orders.py:271-289`
- `append_audit` records the verify result; writes the entry regardless. `_load_seen_nonces` then adds the nonce of a *failing* entry to the seen set — burns a nonce on failed sign calls.
- **Patch**: raise on verify-false in `append_audit`, OR partition into `audit_log.jsonl` (verified) and `audit_log_failed.jsonl` (unverified, no nonce-burn).
